# QL7 Support DM Contract

QL7 Support is a reserved system actor inside the existing Quantum Messenger DM contour.

It is not a normal user account and it does not own a regular profile, wallet, posts, topics, followers, or block/report/follow actions.

## Identity

- Canonical ID: `ql7-support`.
- Avatar: `/ql7/ql7support.png`.
- Actor module: `lib/ql7-support/systemActor.js`.
- The ID is reserved in profile nickname save/check routes and cannot be claimed by a user.

## Delivery

- Incoming support messages are server-only and created through `lib/ql7-support/server.js`.
- Client requests cannot spoof `senderId=ql7-support`; `app/api/dm/send/route.js` rejects that path.
- QL7 Support uses existing DM storage, indexes, unread counters, dialog sorting, and background push.
- Support events must carry a deterministic `dedupeKey`; dedupe state is Mongo primary in `ql7_support_message_dedupe`.
- Redis remains only an existing realtime/push accelerator, not durable support state.
- There is no public browser API for creating system support messages or running broadcasts.
- Long-running time events are planned by `lib/ql7-support/scheduler.js`; scheduler runners are server-side helpers, use Mongo primary reads, and default to dry-run planning.

## User Replies

- Users may reply to QL7 Support with text only.
- Media, voice, video, stickers, VIP emoji, gifts, block, follow, and report actions are disabled for support threads.
- User text replies are bridged to the existing support email transport by `sendQl7SupportEmailBridge`.
- User text replies are classified server-side by `classifyQl7SupportRequest` into broad operational topics such as QCoin, VIP, Ads, moderation, security, account, media, technical, and general requests.
- QL7 Support keeps Mongo-primary request context in `ql7_support_user_requests`; the first message on a topic receives a premium acknowledgement, while follow-up messages on the same topic receive a contextual "already in review" acknowledgement.
- Auto-replies are generated by `buildQl7SupportAutoReply`, localized to the user's selected language, and delivered as normal support DM messages through the same unread and push contour.
- The shared email transport is `lib/supportEmailTransport.js`; `app/api/contact/route.js` and QL7 Support replies both use it, so no second mailer contour exists.

## UI

- Support rows and thread headers render the official support badge.
- Clicking the support avatar or nickname opens `Ql7SupportPopover`, not a normal profile popover.
- The support composer renders text-only controls.
- The DM dialog list always exposes an official `QL7 Support` launcher above the first dialog with a dedicated impulse rail. This lets the user reopen/create the support thread even after deleting local support history.
- Opening an empty or locally deleted support thread calls the normal `/api/dm/thread` route. The server creates a real localized `support_thread_open` greeting from QL7 Support and returns the refreshed thread; the client does not fake that message.
- Translations are required in all supported UI languages: `en`, `ru`, `uk`, `es`, `tr`, `ar`, `zh`.

## Event Templates

`lib/ql7-support/templates.js` owns localized server message templates for:

- welcome;
- QCoin credit;
- VIP activation, expiry warnings, and expiry;
- Ads activation, weekly metrics, expiry warnings, and final summary;
- reports, report threshold, post removal, media lock, rules warnings;
- broadcasts and critical security notices.
- first support-thread openings.

New QL7 Support event integrations must call `deliverQl7SupportEvent` only after the canonical domain operation is confirmed server-side.

## Confirmed Domain Entrypoints

- Registration/wallet session success: `app/api/wallet-session/route.js` calls `notifyQl7Welcome`.
- QCoin confirmed credit: `app/api/qcoin/topup/webhook/route.js` calls `notifyQl7QcoinCredited` only after invoice save and ledger event save.
- VIP activation: `lib/subscriptions.js` calls `notifyQl7VipActivated` through the shared `setVip` contour.
- Ads package activation: `lib/adsCore.js` calls `notifyQl7AdsActivated` after package persistence.
- Forum report/moderation: `app/api/forum/report/route.js` calls report, threshold, removal, media-lock, and rules-warning notifications after the canonical report result.
- VIP expiry, Ads weekly metrics, Ads expiry, and Ads final summaries are server-side scheduled through `lib/ql7-support/scheduler.js` with deterministic subjects and dry-run default.

## Broadcast Safety

- `lib/ql7-support/broadcast.js` is the only broadcast runner.
- Broadcast dry-run is the default.
- Actual send requires `dryRun: false` from trusted server code.
- Broadcast concurrency is bounded and uses existing DM delivery, unread, and push.
- There is no browser route for full broadcast.
- The only DM-triggered broadcast command is a guarded message sent to QL7 Support itself.
- DM-triggered broadcast is disabled unless `QL7_SUPPORT_DM_BROADCAST_ENABLED` is truthy.
- DM-triggered broadcast requires exact token match with `QL7_SUPPORT_DM_BROADCAST_TOKEN`.
- Optional `QL7_SUPPORT_DM_BROADCAST_PREFIX` overrides the command phrase; otherwise the accepted command prefix is `Admin`.
- Optional `QL7_SUPPORT_BROADCAST_ADMIN_IDS` can restrict the command to known admin account identifiers.
- A valid command is parsed, authorized, and executed server-side before normal DM persistence, so the raw token command is not saved to the conversation and is not bridged to email.
- Recipients are discovered from Mongo primary ecosystem collections and delivered through the existing official support DM contour.

Example command when enabled:

```text
Admin <QL7_SUPPORT_DM_BROADCAST_TOKEN> <message text for all ecosystem participants>
```
